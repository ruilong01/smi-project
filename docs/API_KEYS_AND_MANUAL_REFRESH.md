# API Keys and Manual Refresh

## 1. Required environment variables

None of the variables below are required for the app or the refresh
pipeline to run - every one has a safe default, and `refresh:incremental`'s
only live network dependency (OpenAlex) is a free, open API that needs no
key at all. This list is generated by inspecting the actual code (grep for
`process.env.` across `scripts/` and `server/`), not assumed.

| Variable | Used in | Required? | Purpose |
|---|---|---|---|
| `OPENALEX_EMAIL` | `scripts/ingestion/config.mjs` | No (defaults to a placeholder) | Identifies requests in OpenAlex's polite pool (fewer rate limits) |
| `OPENALEX_PAGES_PER_QUERY` | `scripts/ingestion/fetchOpenAlex.mjs` | No (default `2`) | Pages fetched per topic query on a full sweep (`fetch:openalex`) |
| `OPENALEX_INCREMENTAL_PAGES_PER_QUERY` | `scripts/ingestion/fetchGlobalUpdates.mjs` | No (default `2`) | Pages fetched per topic query on an incremental refresh |
| `AI_CURATION_API_URL` | `scripts/ingestion/aiCuration/config.mjs` | No | Future AI image-curation endpoint - unset means `curate:images` no-ops (every candidate stays "pending") |
| `AI_CURATION_API_KEY` | same | No | Auth for the AI curation API above |
| `AI_CURATION_MODEL` | same | No | Model name passed to the AI curation API |
| `AI_CURATION_MIN_SCORE` | same | No (default `60`) | Score threshold below which a curated image is rejected |
| `API_PORT` | `server/server.mjs` | No (default `8787`) | Port the backend API listens on |
| `API_HOST` | `server/server.mjs` | No (default `127.0.0.1`) | Host the backend API binds to (`0.0.0.0` behind nginx on Lightsail) |
| `ADMIN_TOKEN` | `server/server.mjs` | No (unset = feature disabled) | Bearer token required to call `POST /api/admin/refresh-data`; that endpoint 404s entirely until this is set |

Not present anywhere in the codebase today: `OPENAI_API_KEY`,
`CROSSREF_MAILTO`, `UNPAYWALL_EMAIL`, `DATA_REFRESH_TIMEZONE`. No AI API is
called during extraction or refresh (`enrich:explanations` is a heuristic/
template generator, not an AI call - see its file header), Crossref DOI
verification (`processRecords.mjs`) doesn't send a mailto parameter today,
and there's no Unpaywall integration or configurable timezone in the code.
If/when one of these is actually wired in, add it to this table and to
`.env.example` at that time - don't add placeholders for integrations that
don't exist yet.

## 2. Creating a local `.env`

```bash
copy .env.example .env
```

Then edit `.env` and fill in whatever you actually want to change (most
users can leave everything blank/default). `.env` is gitignored - it will
never be committed. Never paste a real key into `.env.example` itself.

## 3. Creating the AWS (Lightsail) `.env`

The server never reads `.env` from git - create it directly on the box,
outside the repo checkout's tracked files:

```bash
ssh <user>@<lightsail-host>
cd /path/to/app
cp .env.example .env
nano .env   # fill in real values for that environment only
```

Only set `ADMIN_TOKEN` on the server if the admin refresh endpoint is
actually meant to be reachable there - leaving it unset keeps
`POST /api/admin/refresh-data` returning 404, which is the safer default
until that flow is deliberately enabled.

## 4. Running a dry run

```bash
npm.cmd run refresh:incremental -- --dry-run
```

This checks configuration (which env vars are set vs default - it never
prints an actual value, so a real contact email or key can't leak into
console/log output), checks which data sources are present
(`data/raw/openalex/`, `data/processed/research-records.json`,
`data/processed/display-records.json`), reads
`data/processed/update-status.json` for the last run's outcome, and prints
the 10 steps a live run would take - all without any network call, any AI
call, or any write to `data/processed/`.

## 5. Running a manual refresh

```bash
npm.cmd run refresh:incremental
```

This calls OpenAlex for real (an incremental window since the last
successful run) and fetches each newly-queued record's own source page(s)
for images. Everything is written to a temp directory first; only if
`verify:data-provenance` and `verify:display-eligibility` both pass against
that temp output, and the record count isn't unexpectedly zero, does it get
promoted into `data/processed/` (see `moveTempIntoPlace` in
`scripts/ingestion/refreshIncremental.mjs`). A failed run leaves the live
`data/processed/*.json` files untouched and records the failure in
`data/processed/update-status.json`.

## 6. Verifying results

After a manual refresh:

```bash
npm.cmd run verify:display-eligibility
npm.cmd run verify:data-provenance
```

Both should pass (they already ran once, inside the temp dir, as a
precondition of promotion - re-running them against the now-live files
confirms the promoted data is still good).

## 7. Checking `update-status.json`

```bash
type data\processed\update-status.json
```

Key fields: `status` (`"success"` or `"failed"`), `lastSuccessfulRefreshAt`,
`lastAttemptedRefreshAt`, `recordsFetched`/`recordsNew`/`recordsUpdated`,
`recordsDisplayEligible`, `errors` (populated only on failure).

## 8. Avoiding committing secrets

- `.env`, `.env.local`, and any other `.env.*` file except `.env.example`
  are gitignored (see `.gitignore`) - only `.env.example`, which must
  contain placeholder values only, is committed.
- Run `npm.cmd run verify:git-clean-for-pr` before opening any PR - it
  flags accidentally-staged secrets/generated files (see
  `docs/DATA_SNAPSHOT_POLICY.md`).
- Never paste a real key/token into a script, a commit message, or this
  documentation.

## 9. If a refresh fails

- The live `data/processed/*.json` files are untouched - nothing users see
  changes.
- Check `data/logs/weekday-refresh.log` (gitignored, local/server only) for
  the step-by-step log and the failure's error message.
- Check `data/processed/update-status.json` - `status: "failed"` and
  `errors` will show what verification step or fetch call threw.
- Common causes: OpenAlex temporarily rate-limiting/unavailable (transient -
  retrying later usually resolves it, see `scripts/ingestion/http.mjs`'s
  retry policy), or `verify:data-provenance`/`verify:display-eligibility`
  catching a real data problem in this run's output (in which case, do not
  bypass the check - the promotion was correctly blocked).
- Re-run `npm.cmd run refresh:incremental -- --dry-run` first to confirm
  configuration and source availability before attempting a real refresh
  again.
